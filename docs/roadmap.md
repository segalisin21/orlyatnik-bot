# Roadmap: бот «Орлятник 21+»

Документ описывает текущее состояние продукта, ближайшие шаги и долгосрочное направление. Живой код и схема диалога: [`dialog-flow.md`](dialog-flow.md), архитектура: [`architecture.md`](architecture.md).

---

## Текущее состояние (сейчас)

### Регистрация и воронка

- Вход: `/start` и приветствия без выбранного мероприятия ведут в **одну ветку — Орлятник 21+** (кнопка «Орлятник 21+»; второе мероприятие в UI отключено флагом `FEATURE_PIZHAMNIK_UI_ENABLED` в [`src/config.ts`](../src/config.ts)).
- Меню после выбора: программа, условия и стоимость, бронь (🔥); **без** кнопки «Сменить мероприятие», пока флаг выключен.
- После блока «Условия и стоимость» — компактная клавиатура (программа / бронь / вернуться в полное меню); длинный текст условий режется под лимит Telegram 4096 символов.
- Анкета: ФИО → город → дата рождения → с кем едет → телефон → комментарий → смена (кнопки из `AVAILABLE_SHIFTS`).
- Согласие на ОПД перед анкетой; подтверждение анкеты; оплата переводом и/или ЮKassa при наличии ключей; чек → уведомление админу → подтверждение оплаты → финальное сообщение с чатом и менеджером.

### Данные и конфигурация

- Участники Орлятника: лист Google Sheets «Участники»; тексты и параметры — лист «Настройки» (перекрывают дефолты из кода).
- Логирование сообщений, кэш участника, FSM-статусы в коде ([`fsm.ts`](../src/fsm.ts), [`sheets.ts`](../src/sheets.ts)).

### Админка

- Рассылка по аудиториям, статистика, правка настроек через бота.
- При выключенном `FEATURE_PIZHAMNIK_UI_ENABLED` в настройках показывается только **Орлятник** (лист «Настройки»); ветка Пижамника в коде и колбэки для старых сообщений сохранены.

### Отключено в интерфейсе (временно)

- **Пижамник:** кнопка выбора, текстовый вход «пижамник» без перехода в ветку (подсказка про только Орлятник), переключение с Орлятника на Пижамник в статусе CONFIRMED.
- **Смена мероприятия:** кнопка скрыта; колбэк `event_change` игнорируется.

Включение обратно: в [`src/config.ts`](../src/config.ts) установить `FEATURE_PIZHAMNIK_UI_ENABLED = true`, задеплоить, при необходимости обновить `ROOT_EVENT_CHOICE_MESSAGE` (двухветочный текст уже в коде под флаг).

## Фаза 1 — стабилизация Орлятника (1–2 недели)

| # | Задача | Критерий готовности |
|---|--------|---------------------|
| 1.1 | Синхронизировать прод-таблицу с актуальными текстами (`START_MESSAGE`, `PROGRAM_TEXT`, `CONDITIONS_TEXT`, `CONSENT_PD_TEXT`, смены, цены) | Менеджер правит только таблицу, бот без релиза показывает те же формулировки, что и согласованный макет |
| 1.2 | Проверить лимиты Telegram на всех шагах (длинные `CONDITIONS_TEXT` из таблицы) | Нет обрезанных/битых сообщений; при необходимости правила разбиения в коде покрывают весь контент |
| 1.3 | Регрессия: оплата переводом, ЮKassa (если включена), чек фото/документ, кнопка админа «Подтвердить оплату» | Полный проход тестового пользователя до `CONFIRMED` |
| 1.4 | Мониторинг: логи webhook/polling, ошибки Sheets | Алерты или ручной чек раз в день на время пика регистраций |

## Фаза 4 — надёжность и масштаб (по мере роста)

- Резервный webhook / health-check сервиса деплоя.
- Версионирование схемы листов (миграции заголовков).
- Резервное копирование таблицы и политика удаления ПДн.
- Нагрузочный тест callback + Sheets при пике одновременных нажатий.

## Технический долг (бэклог без жёстких сроков)

- Единая сборка CI с `typescript` в `devDependencies`, автоматический `tsc` на PR.
- Типизированные ключи настроек вместо строковых Record.
- Вынести `ROOT_EVENT_CHOICE_MESSAGE` в таблицу или env по желанию продукта.
- Клиентские E2E не в Telegram: хотя бы юнит-тесты на `splitTelegramMessage` и парсинг конфига.
